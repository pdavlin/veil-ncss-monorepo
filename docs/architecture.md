# Architecture — Veil + NCSS token system (Phase 2)

Scope: how the three token packages compose into a single CSS surface, and the rules each layer enforces.

## Layer strategy

`packages/tokens-base/dist/index.css` declares the cascade layer order once:

```css
@layer reset, tokens, base, components, utilities;
```

- `reset` — modern CSS reset, owned by `tokens-base`.
- `tokens` — custom properties (scales, aliases, misc, brand overrides). Both `tokens-base` and brand layers emit into this.
- `base` — typographic and document defaults. Phase 3.
- `components` — component-scoped styles. Phase 3.
- `utilities` — atomic helpers. Phase 3 if needed.

Brand layers (`tokens-veil`, `tokens-ncss`) do NOT re-declare `@layer`. Their selectors land in `tokens` by virtue of import order — base before brand.

## Brand scoping

Brand CSS is scoped to `:root, :root[data-brand="<brand>"]`. The compound `:root[data-brand="..."]` form (specificity `0,2,0`) deliberately outranks a bare `:root` (`0,1,0`) so that when both brand layers load simultaneously, the attribute selector wins instead of source-order. Two consequences:

1. **Single-brand sites import only one brand layer.** Bare `:root` matches and the brand attribute is unused.
2. **The validation harness imports both.** With `data-brand="veil"` set, Veil's `:root[data-brand="veil"]` beats NCSS's bare `:root` by specificity. Toggling `data-brand` on `<html>` flips which compound selector matches.

Bare `:root` alone is not enough — when both layers load, two equal-specificity `:root` rules tie and the last-loaded wins for every property. That's why the compound form is required.

## Alias contract

Aliases declared in `packages/tokens-base/config/aliases.ts` are the only color/font surface that downstream components are allowed to use.

- Brand layers **may** override existing aliases.
- Brand layers **must not** introduce new aliases. Extending the surface means editing `tokens-base` and republishing.
- Brand layers **may** expose raw palette tokens (e.g. `--color-veil-ink`) for use *inside* the brand layer's own emissions. Components must not reference them. Lint will enforce this in Phase 3.

This keeps the component surface stable across brand swaps.

## Fluid scales

Type and space scales are generated by `utopia-core` from `packages/tokens-base/config/scales.ts`. At viewport `minWidth` the computed font-size equals `minFontSize`; at `maxWidth` it equals `maxFontSize`. In between, `clamp()` linearly interpolates.

Step labels use the size-relative form (`step--1`, `step-0`, `step-1`, …) and the semantic aliases (`--text-body`, `--space-section`) point at them.

## Contrast resolution

`packages/_lib/src/contrast.ts` runs WCAG 2.2 ratio math. For each documented pairing:

- Already passing? `no-change-needed`.
- Foreground unlocked? Shift it in OKLCH toward higher contrast until the pair clears the AA bar (4.5:1, or 3:1 if `large`).
- Foreground locked, background unlocked? Adjust the background.
- Both locked? Surface as `manual-override`. Build fails until the engineer resolves it.

The Veil build writes its decisions to `docs/color-decisions.md` for client review.

## Hard rules

1. A pair listed as "valid" in `model.json` MUST pass AA. The build re-checks every pair against the final emitted values and exits non-zero otherwise.
2. Brand layers must emit a `:root, [data-brand="<brand>"]` selector for every alias they override.
3. Base aliases must declare a fallback value sufficient to render a default (unbranded) page.

## Breakpoint names

| Variable | Value | Intent |
|---|---|---|
| `--bp-sm` | 480px | phones in landscape |
| `--bp-md` | 768px | tablets |
| `--bp-lg` | 1024px | small desktops |
| `--bp-xl` | 1280px | large desktops |
| `--bp-2xl` | 1440px | maximum viewport for fluid scale |

Container query units are deferred to Phase 3 (they belong with component scopes).
