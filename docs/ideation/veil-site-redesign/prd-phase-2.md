# PRD: Veil + NCSS Site System — Phase 2

**Contract**: ./contract.md
**Phase**: 2 of 4
**Focus**: Convert Phase 1's raw tokens into a brand-themeable design system. Output is two packages: `tokens-base` (shared) and `tokens-veil` (brand layer).

## Phase Overview

Phase 2 takes the curated tokens from Phase 1 and rebuilds them as a coherent, responsive, brand-themeable system. The output splits into two packages:

- **`tokens-base`**: brand-agnostic. Fluid type and space scales (Utopia), radii, motion, breakpoints, semantic token *aliases* with default values. Both sites consume this verbatim.
- **`tokens-veil`**: brand layer. Veil's color palette, font-family choices, brand-specific overrides for any semantic aliases that need it. Imported *after* `tokens-base` so the cascade wins.

The third package, `tokens-ncss`, gets a stub in this phase (placeholder colors/fonts) just to prove the brand layer swaps cleanly. The real NCSS brand work happens in Phase 4 when we have NCSS's actual identity in hand.

Type and space become Utopia-style fluid scales clamped between minimum and maximum viewport widths. Colors get reviewed against WCAG 2.2 AA and adjusted when they fail — adjustments live in the brand layer, not the base. The heavy decision in this phase is contrast resolution for the Veil palette. Every failing color pair from Phase 1 gets a documented adjustment, and the diff between original and accessible values is the artifact the client signs off on before Phase 3 begins.

This phase is sequenced second because it cannot exist without Phase 1's inventory, and it must exist before Phase 3 can build components that consume only semantic tokens.

## User Stories

1. As the implementing engineer, I want a `tokens-base` package that exposes every brand-agnostic primitive (fluid scales, radii, motion, breakpoints) as CSS custom properties, so both sites consume the same scale system without duplication.
2. As the implementing engineer, I want a `tokens-veil` package that exposes Veil's color palette and fonts as CSS custom properties, plus any semantic alias overrides, so swapping `tokens-veil` for `tokens-ncss` changes branding without touching components.
3. As the implementing engineer, I want fluid type and space scales generated by Utopia (or an equivalent clamp-based generator), so layouts breathe with viewport width without a forest of media queries.
4. As the implementing engineer, I want a validation harness that toggles between Veil and NCSS-stub themes at runtime, so the brand-swap property is provable before Phase 3 starts.
5. As the client (via the engineer), I want a side-by-side diff of original vs. accessibility-adjusted Veil brand colors so I know exactly what changed and why before we build pages with the new palette.

## Functional Requirements

### Package Architecture

- **FR-2.1**: Create `packages/tokens-base` as a publishable npm-style package (private to the monorepo).
- **FR-2.2**: Create `packages/tokens-veil` consuming `tokens-base` and emitting only brand-layer CSS.
- **FR-2.3**: Create `packages/tokens-ncss` as a stub: same shape as `tokens-veil` but with placeholder colors (e.g. shifted hue) and a system font stack, sufficient only to prove the brand swap works.
- **FR-2.4**: Each package emits a single `dist/index.css` entrypoint that re-imports its constituent files.

### Fluid Scales (tokens-base)

- **FR-2.5**: Define minimum and maximum viewport widths (e.g. 320px → 1440px) as the clamp range for fluid scales.
- **FR-2.6**: Generate a fluid type scale using `utopia-core`, with steps covering body, small, large, and the heading levels observed in Phase 1.
- **FR-2.7**: Generate a fluid space scale (3xs through 3xl, or similar) with the same clamp approach.
- **FR-2.8**: Output fluid scales in `tokens-base/dist/scales.css` as CSS custom properties on `:root`.

### Semantic Token Aliases (tokens-base)

- **FR-2.9**: Define semantic aliases for color (`--color-surface-default`, `--color-surface-brand`, `--color-text-default`, `--color-text-on-brand`, `--color-border-subtle`, etc.) with **fallback** values in `tokens-base`. Brand layers override.
- **FR-2.10**: Define semantic aliases for typography (`--font-sans`, `--font-display`, `--text-body`, `--text-h1`..`--text-h6`, `--leading-tight`, `--leading-normal`) with fallbacks in `tokens-base`.
- **FR-2.11**: Document the alias contract in `packages/tokens-base/README.md` — what each alias means and what components are allowed to reach for.

### Brand Layer (tokens-veil)

- **FR-2.12**: For each color pair flagged in Phase 1's `a11y-issues.json`, evaluate the contrast ratio against WCAG 2.2 AA thresholds (4.5:1 for normal text, 3:1 for large text and UI components).
- **FR-2.13**: Where a pair fails, adjust the smaller-surface token (typically the text color, not the brand background) until the pair passes, documenting the original and adjusted values.
- **FR-2.14**: Where adjustment changes a brand color the client will notice, flag for client review before locking.
- **FR-2.15**: Emit `tokens-veil/dist/index.css` containing Veil's palette and any semantic alias overrides as CSS custom properties on `[data-brand="veil"]` and `:root` (so import-only theming works for single-brand sites, and `data-brand` attribute switching works in the validation harness).
- **FR-2.16**: Produce `color-decisions.md` showing every original→adjusted pair with hex values, contrast ratios before/after, and the rationale.

### Breakpoints and CSS Architecture

- **FR-2.17**: Define a small set of named breakpoints (mobile, tablet, desktop, wide) as CSS custom properties and container query units where useful, in `tokens-base`.
- **FR-2.18**: Establish a CSS layer strategy (e.g. `@layer reset, tokens, base, components, utilities`) and document it in `architecture.md`. Brand layer imports into the same `tokens` layer, so cascade order within the layer matters and must be enforced by import order.
- **FR-2.19**: Produce a CSS reset (modern-normalize or hand-rolled) scoped to the reset layer, in `tokens-base`.

### Validation

- **FR-2.20**: Provide a validation harness page that toggles between `data-brand="veil"` and `data-brand="ncss"` at runtime, renders every fluid scale step, and renders every documented color pair under both brands.
- **FR-2.21**: Automate contrast checking across all token pairings (both brands) via a script that fails CI if any documented "valid" pair drops below AA.

## Non-Functional Requirements

- **NFR-2.1**: Token CSS is consumable by any tool — no preprocessor, no JS runtime. Just `@import` or `<link>`.
- **NFR-2.2**: Total token CSS payload (base + one brand) stays under 12KB unminified. If larger, the scale is doing too much.
- **NFR-2.3**: Token files are human-readable. Names like `--color-surface-default`, not `--c1`.
- **NFR-2.4**: All Veil color adjustments preserve the spirit of the brand — same color family, same typographic voice. Drift is documented, not arbitrary.
- **NFR-2.5**: The brand layer must not reach into `tokens-base` internals beyond the documented semantic aliases. No `--space-l` overrides in a brand layer.
- **NFR-2.6**: Switching brand layers in a consuming page (changing one `<link>` or `data-brand` attribute) must never cause layout shift beyond what color/font changes naturally produce.

## Acceptance Criteria

- [ ] `tokens-base` and `tokens-veil` packages each expose `dist/index.css` consumable via `<link>` or `@import`.
- [ ] A plain HTML page that imports `tokens-base` + `tokens-veil` (in that order) exposes every documented semantic alias with Veil's values.
- [ ] Swapping `tokens-veil` for `tokens-ncss` in the same page renders the same layout in NCSS-stub colors and fonts without altering structure.
- [ ] Fluid type and space scales clamp correctly between the defined viewport min and max.
- [ ] Every color pair documented as "valid" in both Veil and NCSS-stub passes WCAG 2.2 AA contrast (verified by automated check).
- [ ] `color-decisions.md` lists every adjusted Veil color with before/after hex and contrast ratios.
- [ ] Client has reviewed and signed off on Veil color adjustments.
- [ ] Validation harness with runtime brand toggle renders all scales and pairings under both brands.
- [ ] CSS architecture (`@layer` strategy, alias contract, brand layer rules) is documented and reviewable.

## Open Questions

- Do we use Utopia's web generator and copy output, or pull in `utopia-core` as a dependency to generate on build? (Lean: dependency.)
- Brand swap mechanism: `data-brand` attribute on `<html>` (runtime-toggleable, good for harness) vs. import-only (one brand per build). The harness needs the attribute; production sites can use either. Plan: support both, default to import-only in production builds.
- How many semantic aliases ship in `tokens-base` vs. let brand layers add? Lean toward shipping a tight, opinionated set in base and letting brands only override.
- Does NCSS need any structural tokens that Veil doesn't, or is the system flexible enough that NCSS-only needs land in `tokens-ncss` cleanly?
