# Implementation Spec: Veil + NCSS Site System — Phase 4

**PRD**: ./prd-phase-4.md
**Estimated Effort**: M (1.5–2.5 weeks elapsed, assuming content is in hand)

## Technical Approach

Phase 4 is intentionally additive. The contract: zero changes to `packages/components` and `packages/tokens-base` source. Everything that's NCSS-specific lives under `packages/tokens-ncss/` and `sites/ncss/`. Any change required outside those paths during Phase 4 signals a Phase 3 leak — fix it in `packages/components/` and treat it as a defect, not as Phase 4 work.

Three workstreams:

1. **Real NCSS brand layer**: replace stub palette and fonts in `packages/tokens-ncss/`, run Phase 2's contrast resolution, get client sign-off via `docs/color-decisions-ncss.md`.
2. **Site scaffold**: copy `sites/veil/` to `sites/ncss/`, swap the brand stylesheet, swap the `data-brand` attribute, replace content. Wire Decap with NCSS collections pointed at the shared OAuth proxy.
3. **Launch**: NCSS-specific `netlify.toml`, DNS coordination, a11y/Lighthouse gates, runbook.

Verification includes re-running the two-brand smoke test from Phase 3 to confirm the system still behaves coherently.

## Implementation Details

### Real NCSS Brand Layer

**Pattern to follow**: Phase 2's `packages/tokens-veil/` exactly.

**Overview**: Replace stub values with real NCSS palette and fonts. Run contrast resolution. Generate decisions doc.

**Implementation steps**:
1. Drop real NCSS font files into `packages/tokens-ncss/dist/fonts/`.
2. Update `packages/tokens-ncss/config/palette.ts` with real hex values.
3. Update `packages/tokens-ncss/config/color-overrides.ts` with any client-protected brand colors.
4. Run `pnpm --filter @veil-ncss/tokens-ncss run build`.
5. Inspect `docs/color-decisions-ncss.md` (generated by the build), share with client.
6. Iterate until client approves.

**Key decisions**:
- Same OKLCH adjustment algorithm as Veil. No new code paths.
- If NCSS exposes a structural need Veil didn't (e.g. additional surface tint), evaluate whether it belongs in `tokens-base` (rare, requires a system-wide review) or as an NCSS-only addition layered on top of the alias contract. Default to the latter.

### Site Scaffold

**Overview**: Copy `sites/veil/` to `sites/ncss/`, swap a small set of files, and verify it builds.

**Implementation steps**:
1. `cp -r sites/veil sites/ncss` (then prune Veil-specific content).
2. Update `sites/ncss/package.json` `name` field to `@veil-ncss/site-ncss`.
3. In `sites/ncss/src/_includes/layouts/base.njk`, change `data-brand="veil"` to `data-brand="ncss"`.
4. In `sites/ncss/src/styles/site.css`, replace the tokens-veil import with tokens-ncss.
5. In `sites/ncss/.eleventy.js`, replace passthrough paths for `tokens-veil` with `tokens-ncss`.
6. Delete `sites/ncss/src/content/**/*.md` Veil content; populate with NCSS content.
7. Update `sites/ncss/src/_data/site.js` with NCSS nav, footer, contact info.

**Key decisions**:
- One-time copy is fine. Don't over-engineer a scaffolding script for two sites.
- If NCSS has page types Veil doesn't, add layouts under `sites/ncss/src/_includes/layouts/` composed from existing shared components. Only invent new shared components if a pattern is genuinely reusable.

### NCSS Decap Configuration

**Pattern to follow**: `sites/veil/src/admin/config.yml`.

**Overview**: Same structure as Veil's Decap config but pointed at NCSS content paths and using NCSS-specific page types.

```yaml
backend:
  name: github
  repo: <org>/<monorepo>
  branch: main
  base_url: https://auth.<engineer>.dev          # same proxy as Veil
  auth_endpoint: auth

media_folder: sites/ncss/src/assets/uploads
public_folder: /assets/uploads

collections:
  - name: pages
    label: Pages
    files:
      - file: sites/ncss/src/content/index.md
        # ... NCSS-specific field config
```

**Key decisions**:
- Path-scoped to `sites/ncss/`. Decap writes only to NCSS content; Veil content is untouched.
- Single GitHub OAuth app, single Workers proxy. Brand owners are added to the GitHub team that has commit access.

### CI Matrix

**Overview**: Extend `.github/workflows/ci.yml` to build and validate both sites independently.

```yaml
strategy:
  matrix:
    site: [veil, ncss]
steps:
  - run: pnpm install
  - run: pnpm -r run typecheck
  - run: pnpm -r run lint
  - run: pnpm -r run build
  - run: pnpm --filter @veil-ncss/site-${{ matrix.site }} run build
  - run: pnpm run a11y:check -- sites/${{ matrix.site }}/_site
  - run: pnpm run smoke:two-brand
```

**Key decisions**:
- Matrix on `site` keeps Veil and NCSS jobs independent. NCSS regressions don't block Veil deploys.
- Two-brand smoke runs once per CI invocation (not per matrix row) since it's the same artifact either way.

## Error Handling

| Error Scenario | Handling Strategy |
|----------------|-------------------|
| Contrast adjustment can't reach AA on a critical NCSS brand color | Surface in `color-decisions-ncss.md` and pause for client decision before moving on. |
| Phase 4 reveals a leak in `packages/components` | Stop Phase 4, fix in components, re-run Phase 3 a11y + smoke + Veil-prod regression check, then resume. |
| NCSS DNS cutover affects an existing site | Coordinate cutover window; runbook documents fast revert via DNS only. |
| Shared OAuth proxy hits Workers free-tier limit due to NCSS adds | Investigate; Cloudflare Workers free tier (100k requests/day) is generous, so this is unlikely. If it happens, consider deduplication or a paid-tier conversation. |

## Rollout Considerations

- **Feature flag**: None. Launch is a DNS cutover for NCSS (or a fresh DNS pointing for greenfield NCSS).
- **Monitoring**: Netlify build logs for the NCSS site; Lighthouse CI as a non-blocking signal.
- **Alerting**: Netlify deploy notifications to engineer email.
- **Rollback plan**: `docs/launch-runbook-ncss.md` covers DNS revert (existing site replaced) or DNS pointing back to nothing/holding page (greenfield).
- **Veil regression watch**: First 48 hours after NCSS launch, check that the OAuth proxy hasn't started misrouting and that Veil's CMS still works for Veil's collection paths.
